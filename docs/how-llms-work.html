How these open-source models actually work

The mental model behind Qwen, why "reading files" isn't a thing the model does, and the two-model architecture we're building around it.

1. The model is just text → text

An open-source model like Qwen is, at heart, a function that takes text in and produces text out. That's the whole interface. It has:

  • no filesystem — it can't open files;
  • no memory between conversations;
  • no agency — it can't decide to "go look something up".

The only way information reaches it is the context window — the block of text you hand it each turn (system prompt + conversation so far + your new message). Whatever's in that block, it "knows". Whatever isn't, it doesn't.

2. "Reading files" is the app's job, not the model's

When you see a coding assistant "read your files", the model isn't doing that. The program around the model (the "harness" — the app) reads the file and pastes its text into the context window. The model just sees text.

Our reading layer works the same way: the model outputs clean Chinese, and app.py (via jieba / pypinyin / CC-CEDICT) adds the pinyin and hover glosses. The model never touches a file.

3. Two ways to shape behavior

There are two completely different levers for changing what a model does — and knowing which is which is the key insight for the whole project:

TeachesStyle / behavior — "respond like an HSK-5 tutor, bilingual, this format"Specific current info — "focus on these 20 words this week"

Rule of thumb: if it changes often or is specific (a curriculum, a word list), it belongs in context, not training. You'd never retrain the model every time the lesson changes.

4. Our fine-tune is knowledge distillation

A big capable model (Claude) writes the training examples; a small model (Qwen) learns to imitate them. That's distillation — teacher → student.

Claude (teacher) ──generates──▶ data/train.jsonl ──trains──▶ Qwen2.5-7B (student) smart, expensive ~900 examples small, yours to run

Crucially, the fine-tune doesn't teach Qwen Chinese — it already knows Chinese from pretraining. It teaches the behavior: HSK-5 level, bilingual format, the correction shape, the encouraging tone. That's why the base-model quality sets the ceiling (why we chose 7B) but the dataset can stay small.

5. The two-model architecture we're building toward

Because the small model can't plan or remember, the design splits responsibilities:

RoleWhoWhy
Coach / plannerClaudeHas reasoning + memory (via files). Discusses goals, tracks progress, writes the plan. Runs occasionally.
Drill instructor / executorQwen (fine-tuned)Small, fast, local, cheap. Does the actual HSK-5 tutoring. Runs constantly.
Shared memorycurriculum.md / progress.mdClaude writes them; the app persists them; Qwen reads them (injected into its prompt). The files ARE the long-term memory Qwen lacks.

This is the standard "planner + specialized executor with a shared state file" pattern. The planner has judgment and memory; the executor is cheap and focused; a plain text file is the bridge between them.

6. Why this matters for the project

Everything that isn't "be a good HSK-5 tutor" lives in the app, not the model — the pinyin/hover reading layer, curriculum injection, flashcard extraction. They're all built from deterministic tools (jieba / pypinyin / CC-CEDICT) around a focused model. The payoff: these features stack up without ever forcing a retrain.
