QA · report

Multi-scenario QA sweep of the HSK-5 tutor

61 real serve-layer exchanges — 5 conversations, 16 Q&A prompts, 3-roll consistency checks — generated headlessly through the exact app.py prompts and gates, then graded against the tutor's own Chinese, correction truth, the bilingual/Chinese-only contracts, driving, level, coherence, and consistency.

Headline: The tutor drives conversation well, stays coherent over long sessions, and honors both language contracts. Its one serious, reproducible flaw is inventing grammar errors in sentences that are already correct — and often "fixing" them into worse Chinese. This is a data gap, not a model-size ceiling. Recommendation: do one data iteration + retrain before considering Qwen-14B.

How this was measured

A headless harness imported app.py (loading the same GGUF the app serves) and pushed simulated students through respond()'s real logic — same system prompts (SYSTEM_PROMPT_APP / conversation_system()), same temperature 0.7, same correction-extraction, enforcement, disambiguation, and translation-fill gates. So every judgment below is about the shipped behavior, not a re-implementation. Generation was fast: 1.8 s mean, 6.3 s max per turn on Metal. A browser pass confirmed the serve-layer features end to end (starters valid, streaming→annotate swap, hover tips, Q&A fills, 聊天 English-input path, no console errors).

The main finding: false corrections on correct input

Whenever a grammatically correct sentence was shown, the tutor frequently fabricated an error and "corrected" it — usually into something ungrammatical.

WhereFalse corrections
聊天 (chat_correct)2 / 6 turns
Q&A correct-sentence probes1 / 2
Consistency: 我每天早上七点起床 ×3 rolls3 / 3 rolls
Correct-sentence presentations, combined6 / 11 (≈55%)

The consistency roll is the clearest proof: one perfectly correct sentence, three rolls, three different invented rules — and roll 3's "fix" is itself wrong Chinese:

student: 我每天早上七点起床。 (correct — "I get up at 7 every morning")
roll 1 → 七点钟 is "missing a measure word" (false — 七点 is standard)
roll 2 → garbled rule about 早上/在, no real fix
roll 3 → "add 了 for habitual actions" → 我每天早上七点起床了 (ungrammatical)
student: 我觉得听力最难，因为说话的速度太快了。 (correct)
tutor: 应该说"说话的声音太快了"… (声音快 is wrong; 速度快 was right)
student: 我已经把作业做完了。 (correct)
tutor: 应该用"完成"不用"做"… 修改后：我已经把作业完成了。 (做完 is completely standard; rule invented)

Root cause — training data, not capability. The correct_sentence task (180 examples) always contains an error to fix. The model never saw a "this sentence is already correct — nothing to change" example, so it learned the prior "if asked to check a sentence, find something to fix." The one time it got a correct sentence right (虽然…但是…) shows the base model can tell; it just wasn't trained to. This is a data iteration fix: add correct-sentence examples whose ideal answer confirms correctness.

Serve-layer consequence. The correction-chip overlap gate (≥60% of the fix's characters reused from the student) can't catch these: a fabricated correction of an already-correct sentence naturally reuses almost all the student's characters, so it passes the gate. Chips fired on 2/6 false corrections in chat_correct — meaning false corrections can flow into the flashcard deck as fix-cards that teach wrong Chinese. The durable fix is upstream (data), not a tighter gate.

Second finding: confabulated grammar rules

Even when the tutor caught a real error and produced a reasonable corrected sentence, the stated reason was often wrong or self-contradictory. Among the 7 corrections issued on genuinely-wrong sentences in chat_errors, ~3 carried a flawed rule:

他比我更高很多。
fix "他比我高很多" ✓, but rule: "形容词后面要加'得'" — contradicts its own fix (which has no 得)
我去过北京去年。
"修改后"：我去过北京去年 (identical — no change); rule says time words go after the verb (backwards)
我把这本书看。
fix "看完" ✓, but calls 看 an 不及物动词 (intransitive) — it's transitive
Q&A: "居然 vs 竟然?"
"居然…常常是好的结果" (false — 居然 is neutral/negative surprise), and "可以互换使用"

This is the one place with a genuine capability signal: producing the right surface correction is a pattern the fine-tune learned; explaining the underlying rule is knowledge, and the 7B confabulates it. But note two mitigating facts: (a) the "no-change fix + backwards rule" cases are downstream of the same must-find-an-error data artifact, and (b) the flaw degrades the explanation, not usually the corrected sentence — less harmful than teaching a fake error. Worth re-measuring after the data fix before treating it as a model-size ceiling.

Third finding: forced target words

The "every 聊天 reply must use a target word or invite one" rule occasionally forces a semantically broken insertion (~2–3 / 37 turns):

感觉必然跟朋友打电话聊天 (必然 "inevitably" is nonsense here)

prompt/capability — the softer "when natural" phrasing was already tried and ignored by the 7B (per DECISIONS 2026-07-10). A 14B would likely satisfy the hard constraint more naturally; a cheaper partial fix is to let the tutor invite usage more and self-insert less.

Recommendation: data iteration first, not Qwen-14B

The dominant, most damaging failure — false corrections at ~55% on correct input — is a training-data gap the model was never shown how to avoid, not a capability ceiling. That's the cheap, high-leverage fix:

  1. Add "already-correct" examples to correct_sentence in gen_data.py: ~30–40% of that task's prompts should be sentences with no error, where the ideal assistant answer confirms correctness and maybe offers an optional polish — never inventing a fault. Teach the "nothing to fix" response explicitly.
  2. Tighten the correction-task instruction to require the stated rule to be consistent with the corrected sentence (targets the confabulated-rule cases that are data-driven).
  3. Retrain (gen_data → Colab A100 → merge → GGUF, ~1h) and re-run this exact harness to measure the new false-correction rate.

Only escalate to Qwen2.5-14B if confabulated grammar rules stay high on genuinely-wrong sentences after the data fix. The 14B (~9GB Q4, fits the Mac, ~2× serve latency → ~4s/turn, same pipeline) is a reasonable fallback and would help most with the one true capability signal (metalinguistic grammar knowledge) and with satisfying the target-word constraint naturally. But jumping now would (a) mask the data bug — a bigger model still shouldn't be trained to always find an error — and (b) cost a retrain cycle either way. Fix the data, re-measure, then decide. The 7B's driving, coherence, and contract adherence are already good; nothing there argues for more size.
