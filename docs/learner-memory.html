Memory of what you've learned

How the tutor can know what's review vs. new — without blowing the context window.

The worry, addressed

"Won't remembering everything I've studied need a huge amount of context?" No — if the app injects a compact summary of your learning state, not your raw conversation history. Dumping every past message would grow forever and eventually overflow. A summary stays small and flat.

The principle (same as everywhere in this project): the model is stateless — the app supplies memory through the prompt. The trick is summarize + retrieve, never dump.

You already built the memory store

The flashcard deck is a record of what you've learned. Each card's SM-2 stats already encode mastery:

Card stateMeansThe tutor should…
high interval / easeyou know it wellnot re-explain it (unless asked)
lapsed / low intervalyou're shaky on itweave it in for review
recently addednew / learningreinforce it

So "what's review vs. new" is already sitting in the deck — no separate tracking system needed.

What the app injects (each turn)

A small block prepended to the system prompt, derived from the deck:

The learner already knows these words — don't re-explain unless asked:
  承担, 既然, 规矩, 安慰 …
Currently reviewing (weave in naturally where it fits):
  妨碍, 承受 …
  • Grammar points are few (a few dozen total) → you can list them all cheaply.
  • Vocabulary is many (hundreds over time) → don't inject them all every turn. Inject a bounded slice: the words relevant to the current message + the ones due for review, or just a count ("you know ~400 words") plus the relevant handful.

The context math (why it's cheap)

A summary block of ~50–100 words + statuses is only a few hundred tokens. Qwen2.5 handles 32K tokens natively (up to 128K). So even a generous learner profile is a rounding error in the budget. What you must avoid is the unbounded thing — pasting the full transcript of every past session. Cap the injected vocab list (e.g. top-N relevant + all due) and it never grows out of control.

Rule of thumb: inject state (a bounded summary), never history (an unbounded log).

Who keeps it updated

  • App-side, deterministic (enough on its own): when you add a word to the deck, when the tutor explains a word, when you get a correction wrong — the app updates the deck / a small profile. No LLM, reliable, free.
  • Optional — the Claude coach: periodically Claude reads your recent sessions + progress and writes higher-level notes to progress.md ("strong on aspect particles, shaky on 把-sentences"). Richer, but needs the coach step from the curriculum architecture.

The one choice to make

  1. Deck-derived only — the app builds the "known / reviewing" summary straight from the flashcard deck. Simple, deterministic, free. Recommended start.
  2. + Claude progress notes — add the coach writing progress.md for higher-level strengths/weaknesses. Richer, more moving parts.
