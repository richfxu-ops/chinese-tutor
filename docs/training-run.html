The Colab training run, exactly

What each stage of train_colab.ipynb does, what QLoRA is actually doing under the hood, and why we convert the result three times before it reaches your Mac.

The one-line version

We take stock Qwen2.5-7B, nudge a tiny set of add-on weights so it behaves like our tutor, then repackage the result into a small file the Mac can run. All the heavy compute happens on Colab because it needs a CUDA GPU.

Why Colab at all

bitsandbytes 4-bit quantization is CUDA-only — it won't run on Apple Silicon. So training happens on a rented GPU; only the final small artifact comes back to your Mac.

What the "full train" step is actually doing (QLoRA)

Three ideas stacked. This is the part specific to fine-tuning LLMs:

1. Load the 7B in 4-bit, frozen

Qwen's ~7.6B weights are quantized from 16-bit (~15GB) down to 4-bit (~4.5GB) so they fit a modest GPU. These weights are frozen — no gradients, they never change. This is the Q in QLoRA: a quantized base.

2. Attach LoRA adapters — the only thing that trains

On each attention + MLP projection we add a pair of small low-rank matrices (rank r=16). Instead of updating 7.6B frozen weights, we only learn these adapters (~40M params, <1% of the model). The frozen weights do the heavy lifting; the adapters learn the adjustment toward tutor behavior. This is the LoRA — low-rank adaptation — and it's why the whole thing fits in memory and trains in an hour instead of needing a cluster.

3. A standard supervised loop — with one twist

For each batch:

  • format the example with Qwen's chat template (system/user/assistant turns → one token sequence);
  • forward pass, cross-entropy loss — but masked to the assistant tokens only (our DataCollatorForCompletionOnlyLM). It's graded on reproducing the tutor's reply, not the system prompt or your question;
  • backprop → gradients flow only into the LoRA adapters → optimizer (paged_adamw_8bit) updates them;
  • 3 passes over the 810 examples, effective batch size 16, LR 2e-4 cosine-decayed.

The result is not a full model — it's a small adapter that says "here's how to bend Qwen toward tutoring".

The three conversions after training

StepWhy it exists
EvalSanity-check the improvement: run the held-out prompts with the adapter off (plain Qwen) and on (the tutor), so you can see before/after.
MergeYou can't conveniently serve "4-bit base + separate adapter" on a Mac. So we reload the base in 16-bit and mathematically add the adapter's deltas into the weights → one standalone model.
GGUF quantizeThat merged model is 15GB and slow on MPS. We convert it to GGUF (llama.cpp's format) and re-quantize to 4-bit (Q4_K_M, ~4.7GB), which runs fast on Apple Silicon via Metal. This file is what app.py loads.

What to watch

Sanity run: the printed loss should trend down across the 20 steps. If it's flat or NaN, the assistant-masking template didn't match Qwen's ChatML — stop, and we fix RESPONSE_TEMPLATE in train.py before spending time on the full run.

Full run: final train + eval loss, then the eval.py report — that's where we actually judge whether the tutor got better.

Time & cost

~30–60 min of Colab GPU time (L4 / A100 comfortable; free T4 works but slower). It spends your Colab compute, not API dollars — the data generation was the only API cost.
