How all the code works

A file-by-file tour of the project, and the one data structure that ties it all together. Read top-to-bottom or jump to a file.

The big picture

The whole system is a pipeline. A big model writes training data, a small model learns from it, and a thin app wraps the small model with helpers. Nothing clever hides in the model — everything user-facing (pinyin, glosses, flashcards) is ordinary code around it.

config.py ← every stage reads this (the hub) │ ▼ gen_data.py ──▶ data/*.jsonl ──▶ train.py ──▶ adapter ──▶ merge.py ──▶ GGUF (Claude writes) (curriculum) (QLoRA) (fold in) (shrink) │ │ eval.py app.py ──▶ you (before/after) (chat + annotate.py + flashcards)
The recurring theme: the model only does "be an HSK-5 tutor." Pinyin, hover glosses, flashcards, audio — all app-layer code (jieba / pypinyin / CC-CEDICT / a bit of JS), never the model. That's why features stack up without retraining.

The thread that ties it together: the chat record

Every training example — and every message at inference — is this shape:

{
  "task": "correct_sentence",          // which of the 6 task types
  "hsk_target": ["承担"],               // the seed word/grammar it was built around
  "messages": [
    {"role": "system",    "content": SYSTEM_PROMPT},   // the tutor persona
    {"role": "user",      "content": "帮我改：我昨天去过北京。"},
    {"role": "assistant", "content": "..."}            // the ideal tutor reply
  ]
}

gen_data.py produces these, train.py learns from them, and app.py reuses the same SYSTEM_PROMPT at serve time — so the model is trained on exactly the behaviour we ask of it later. Keep that record in mind and the rest follows.

1 · config.py — the hub

The single source of truth. Every script imports it, so nothing drifts. It holds:

  • Paths — where data, the adapter, the merged model, and the GGUF live.
  • Model ids — BASE_MODEL = "Qwen/Qwen2.5-7B-Instruct" (the student) and TEACHER_MODEL = "claude-sonnet-5" (writes the data).
  • SYSTEM_PROMPT — the tutor persona (HSK-5, bilingual, no inline pinyin, correction format). Used in every training example and at inference.
  • TASKS — a list of six TaskSpec objects (explain_word, use_in_sentence, correct_sentence, translate, example_dialogue, grammar_point). Each says how many examples to make, whether to inject a vocab word or grammar point, and a plain-English instruction for the teacher. Keeping prompts as data keeps gen_data.py thin.
  • TRAIN — a TrainConfig dataclass with all the QLoRA knobs: LoRA rank/alpha, 4-bit settings, learning rate, batch size, epochs.
Want to change the tutor's behaviour, the dataset size, or a hyperparameter? It's all here — one file.

2 · gen_data.py — Claude writes the curriculum

Turns the seed word lists into ~900 tutoring examples by calling Claude. The flow:

  1. Load seeds (load_seed) — read hsk5_vocab.txt / hsk5_grammar.txt, skipping comments.
  2. Plan batches (make_item_batches) — for each task, pick its target words (shuffled, cycling if needed), chunk into groups of 5. Each group = one API call; each word = one example, so coverage is even.
  3. Build the prompt (build_teacher_prompt) — assemble what we send Claude: the task instruction + the tutor persona + the 5 target words + "return a JSON array".
  4. Call Claude, in parallel (generate_batch via a ThreadPoolExecutor) — ~180 calls, 8 at a time. response_text pulls text out of the reply (skipping Sonnet 5's thinking blocks — the bug we hit and fixed). extract_json_array is tolerant of stray formatting; a bad batch is retried, then dropped — one failure never kills the paid run.
  5. Wrap + split (to_records, then run) — fold each {user, assistant} pair into the chat record above, hold out 10% per task for eval, shuffle, and write data/train.jsonl + data/eval.jsonl.

CLI: --dry-run (assemble prompts, no API, free), --limit N (cheap smoke test), --workers.

3 · train.py — the QLoRA fine-tune

Runs on Colab (needs a CUDA GPU). Three helper pieces + a main:

  • build_bnb_config() — the 4-bit quantization config (nf4, double-quant). This is the Q in QLoRA: the 7B loads at ~4.5GB instead of ~15GB.
  • build_lora_config() — the LoRA adapters (rank 16) attached to the attention + MLP projections. Only these ~40M params train; the 7B stays frozen.
  • main() — loads the data as a "conversational" dataset, loads the 4-bit model, and hands both to TRL's SFTTrainer.

The important subtlety is the loss mask: a DataCollatorForCompletionOnlyLM with the response template "<|im_start|>assistant\n" makes the loss count only the tutor's reply, not the system prompt or your question. It trains 3 epochs and saves a small adapter (tens of MB) — not a whole model.

(For the deeper mechanics — 4-bit + LoRA + the masked loop — see the training-run doc.)

4 · eval.py — did it actually get better?

The before/after check. Clever bit: it loads the base 7B once, attaches the adapter, and toggles it:

with model.disable_adapter():   # adapter OFF  → plain Qwen ("before")
    base_out = generate(...)
tuned_out = generate(...)        # adapter ON   → the tutor ("after")

It runs both on the held-out prompts, writes a side-by-side eval_report.md, and — with --judge — asks Claude to score both on a 4-axis rubric (level-fit, correctness, task, format) and tally which wins.

5 · merge.py + the GGUF step — package for the Mac

QLoRA leaves you with "4-bit base + separate adapter", which is awkward to serve locally. merge.py reloads the base in 16-bit and folds the adapter's weights in (merge_and_unload()) → one standalone model. The notebook then converts that to GGUF and re-quantizes to 4-bit (Q4_K_M, ~4.7GB), which llama.cpp runs fast on Apple Silicon. That file is the deliverable you download.

6 · annotate.py + get_cedict.py — the reading layer

This is the "pinyin over every character + hover gloss" magic — and it's pure app-side code, no model. annotate(text):

  1. finds the Chinese runs in the text (a regex; English passes through untouched);
  2. segments each run into words with jieba (Chinese has no spaces);
  3. gets per-character pinyin with pypinyin and an English gloss from the bundled CC-CEDICT dictionary;
  4. emits HTML: <ruby>休<rt>xiū</rt></ruby> for pinyin-above, wrapped in a <span title="…"> for the native hover tooltip.

get_cedict.py just downloads that dictionary once (with a TLS fix for macOS). Because pinyin comes from a library, it's always correct — the model never has to guess it.

7 · app.py — the demo

A Gradio chat that serves the merged GGUF locally via llama-cpp-python (Metal on your Mac). The one design trick worth knowing:

Two copies of the conversation. The model always sees the raw text (feeding it our HTML back would poison its context); the chat panel shows the annotated HTML (pinyin + hover). That's why there's a raw_state alongside the visible chatbot.

On each turn it prepends the SYSTEM_PROMPT, calls the model, then runs the reply through annotate() before display. It also has the six starter-example buttons.

8 · web/flashcards.html — the SM-2 review widget

A self-contained page (HTML + CSS + JS, no server). The deck lives in the browser's localStorage. schedule(card, rating) is the SM-2 spaced-repetition algorithm: rate a card Again/Hard/Good/Easy and it sets the next due date via an interval + ease factor. Plus 🔊 browser text-to-speech. It'll become a tab inside app.py at ship, so "click a word in chat → it's in your deck" works with no export.
